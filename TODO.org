* TODO [#B] Let prose motion fall through the comment↔code boundary
  =haskell-ts--forward-sentence= runs prose motion over one =text= node's
  text alone, so reaching a comment's first/last sentence and moving
  again stops at the node boundary rather than continuing into the
  surrounding code (the scratch buffer's edge signal is now caught, not
  propagated — see its docstring).  Confinement was only ever a means to
  get text objects working inside comments, not a goal; ideally plain
  =M-e=/=M-a= would cross into code.  Risky, though: the fix must not let
  text objects (=evil= =d a s=, =a p=) spill past the comment, and it pokes
  the same =evil= bounds machinery that has already broken three times
  (see =NOTES.org=).  Sketch: on the caught buffer-edge signal, if arg is
  unspent, delegate the remainder to =treesit-forward-sentence= from the
  mapped-back real boundary — verify against the evil text-object tests
  first.

* TODO [#B] =backward-sexp= stalls at the end of the last top-level binding
  With point exactly at the end of the file's last top-level binding
  (its =bind= node's end, before the trailing newline), =backward-sexp=
  does not move: treesit reports "at top level" there because the node
  ends *at* point, so =treesit-thing-prev= finds nothing before it and
  there is no enclosing parent.  Better than the pre-fix behaviour (it
  used to run back to =point-min=; see =haskell-ts-test-sexp-backward-top-level=),
  and =backward-sexp= between two top-level bindings steps correctly, so
  this is only the single boundary position.  A fix would need to step
  back over a =bind= whose end coincides with point rather than relying
  on treesit's prev/parent lookup; low priority.
