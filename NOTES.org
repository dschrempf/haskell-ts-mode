* Deleting a sentence that spans a comment continuation line swallows that line's marker
  Fixed for paragraph detection and sentence bounds proper (see CHANGELOG.org,
  "Unreleased" — =haskell-ts--forward-sentence= now dedents each line before
  running prose motion, so a marker-only line reads as a real paragraph break).
  Not fixed: a *single* sentence that itself wraps onto a continuation line
  still has that line's =--=/=-- =-marker included when the region between the
  sentence's start/end points is deleted (e.g. =d a s=/=kill-sentence=) — motion
  only returns a point, and the real buffer text between two points is whatever
  sits there, marker included. Fixing that needs the deletion commands
  themselves to be marker-aware, not just motion. evil-textobj-tree-sitter
  (https://github.com/meain/evil-textobj-tree-sitter) would not help here or
  with the above: tree-sitter treats a comment's/string's whole body as one
  opaque leaf node, with no notion of sentences/paragraphs inside it — that's an
  inherently textual problem, which is why this mode reimplements it manually
  rather than deferring to the grammar.

* =evil='s =}=/={= do not confine to a comment glued to code, unlike =a p=/=i p=
  =d a p=/=v a p= (=evil-select-an-object=/=evil-select-inner-object=) are fixed
  (see CHANGELOG.org, "Unreleased") by narrowing the buffer to the comment
  before =evil= computes the object, when the comment has no blank line
  separating it from surrounding code.  =evil-forward-paragraph=/
  =evil-backward-paragraph= (the =}=/={= motions) go through the same
  =forward-paragraph=/=start-of-paragraph-text= primitives, also advised to clamp
  to such a comment -- but =evil-forward-end=, which those two motions call,
  nudges point one character past the comment's end (or before its start)
  *before* searching, specifically so that starting already at a thing's
  boundary still counts as progress.  That one-character nudge lands outside
  the comment node, so the clamp never triggers and =}=/={= still run straight
  through into the surrounding code.  Not fixed, since the reported bug was
  about the text object, not the motion; would need advising
  =evil-forward-paragraph=/=evil-backward-paragraph= (or =evil-forward-end=/
  =evil-backward-beginning=) themselves.

* Confining =evil='s paragraph object to a glued comment has been fragile
  Three bugs in a row in the same feature (see CHANGELOG.org, "Unreleased",
  the =v a p=/=d a p= entries): each fix for one reported case (both sides
  glued; one side glued; a probe from *code* landing in an *unrelated*
  comment elsewhere) broke, or narrowly avoided breaking, another, because
  `evil-select-an-object'/`bounds-of-thing-at-point' repeatedly re-probe
  =forward-paragraph=/=start-of-paragraph-text= from intermediate positions
  during a single object computation, and those positions are not
  predictable from the call site.  The current fix (narrow once, for the
  whole call, based on the object's actual start; suppress the separate
  per-call clamp for that whole call so it can never fire on an unrelated
  probe) resolves the three known cases and passes an explicit stress test
  (chained comment+function pairs, point in code vs. in comment, on either
  side), but the pattern of "another `evil' internal detail invalidates the
  fix" recurring three times in a row is reason enough to distrust that
  this is the last one.  If another case surfaces, resist patching a fourth
  special case on top of the last three; re-derive from how `evil' actually
  computes bounds (verified empirically each time above, not from
  documentation, since `thingatpt'/`evil' source is not distributed as
  plain text on this system) rather than reacting to the new symptom.

* =backward-sexp= at the end of a =where=/=let= block's last local binding
  Found while fixing the coarse =forward-sexp= over local bindings (see
  CHANGELOG.org, "Unreleased"); pre-exists that fix (reproduces
  identically on =main= before it) and is a =backward-sexp= boundary
  quirk, not a =local_binds= exclusion one, so left untouched here.
  With point at the exact end of a =where=/=let= block's last binding:
  if that binding is also the last thing in the file, =backward-sexp=
  does not move at all — the same "at top level" stall as the
  already-tracked top-level case (=TODO.org=, "=backward-sexp= stalls at
  the end of the last top-level binding"), just one level down.  But if
  the enclosing top-level binding is *not* the last one in the file
  (more top-level code follows the =where=/=let= block), =backward-sexp=
  instead swallows the *entire* enclosing binding — =where='s keyword,
  header, and all local bindings — in one step, rather than stepping
  back over just the last local binding.  Likely the same underlying
  mechanism as the tracked stall (treesit's prev/parent lookup at a
  position where several node ends coincide), worth triaging together
  with that TODO rather than as a separate fix.
